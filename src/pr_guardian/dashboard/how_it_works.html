How It Works

The Review Pipeline

Every PR flows through a 6-stage pipeline. Deterministic gates run first, then AI agents analyse what matters, and a rules-based engine makes the final call.

Webhook Received

GitHub or Azure DevOps fires a PR event

0

Discovery

Parse the diff, detect languages, map the security surface, compute blast radius, and load historical hotspot data.

diff Parse & chunk lang Detect stack sec Surface map blast Radius calc hist Hotspot load
1

Mechanical Gates

HARD BLOCK ON FAIL

Deterministic checks run in parallel. If any fail, the PR is immediately blocked — no agents run.

Universal checks

Gitleaks secrets Semgrep static analysis PII Scanner personal data API Contract stability Migration Safety DB changes

Language-conditional

npm audit JS / TS pip audit Python
2

Triage

Classify the change's risk tier based on what was touched, then select which AI agents need to run.

Trivial Docs, config, generated files
0 agents
Low Minor changes, no risk flags
1 agent
Medium Data layer, API boundary, hotspot
2-3 agents
High Security surface, wide blast radius, cross-arch
6 agents

Amplifiers: Cross-stack changes, >3 languages, SQL/Terraform/Dockerfile, elevated repo risk class, or custom path-risk rules can bump the tier up.

3

AI Agents

Selected agents run in parallel. Each returns findings with severity, certainty, and evidence. Prompts are fully editable on the Prompts page.

Security & Privacy 3.0x

Vulnerabilities, data handling, auth flaws

Coverage gaps, assertion quality, test hygiene

Architecture & Intent 2.0x

Design violations, layer boundaries, consistency

Performance 1.5x

Regressions, N+1 queries, bottlenecks

Hotspot Focus 1.5x

Extra scrutiny on historically problematic files

Code Quality 1.0x

Clarity, observability, logging, error handling

Decision Engine

Deterministic, rules-based. Scores every finding, validates certainty claims against evidence, then applies the decision matrix.

Validation

CONDITIONAL

If findings remain, an adversarial validator challenges each one to reduce noise. Findings without sufficient evidence get their certainty downgraded.

Auto-approve

Score ≤ 4.0, trust tier permits, no overrides

Human Review

Score 4.0 – 8.0, or override triggered

Reject

High-confidence findings with fix suggestions

Hard Block

Mechanical failure or score ≥ 8.0

Certainty Validation

Agents claim a certainty level for each finding. The decision engine validates these claims against actual evidence before scoring.

Detected

Requires ≥2 signals: pattern match + CWE, concrete suggestion, full context, or cross-references

Suspected

Requires ≥1 signal: pattern match, full context, or concrete suggestion

Uncertain

Default fallback when evidence is insufficient

Trust Tiers

Orthogonal to risk. Determined by path-based config rules, then escalated one-way if security-sensitive keywords appear in findings.

AI Only

AI decides autonomously. Auto-approve allowed.

Spot Check default

Auto-approve allowed. Optional human review requested.

Mandatory Human

Blocks until any human approves.

Human Primary

Blocks until a designated reviewer group approves.

Escalation keywords: auth, permission, token, credential, secret, encryption, crypto, rbac, role, privilege, session, injection, csrf, xss, cors, oauth, jwt, certificate, password, api_key

Override Rules

Even if the score says auto-approve, these conditions force a human review.

Any agent returns FLAG_HUMAN verdict
≥1 DETECTED + MEDIUM/HIGH/CRITICAL finding
≥3 SUSPECTED findings
PR adds new external dependencies
Trust tier is Mandatory Human or Human Primary
Target branch: main, master, release/*

Noise Reduction

Findings are always scored in full, but lower-risk tiers suppress noisy results in the PR comment.

Low tier: suppress all LOW severity findings
Medium tier: suppress LOW severity + UNCERTAIN certainty
High tier: show everything
