PR Guardian v0.1

Guide

PR Guardian reviews pull requests automatically using specialist AI agents. It reads the diff, classifies the risk, runs the right agents in parallel, scores the findings, and posts a decision — all within seconds of a PR being opened.

You can customise its behaviour per-repository with two optional files committed to the repo root.

Repository files

Both files are optional. If neither exists, PR Guardian uses sensible defaults for everything.

review.yml YAML

Structured configuration that controls how PR Guardian behaves for this repository. Place it at the repo root. Values are merged on top of the service defaults — you only need to include the keys you want to override.

Key sections

Section What it controls
repo_risk_class standard | elevated | critical — raises the baseline risk for all PRs in this repo
auto_approve Enable/disable auto-approval, set allowed and blocked target branches
thresholds Score boundaries for auto-approve (≤4), human-review (≥4), and hard-block (≥8)
weights How much each agent's score matters in the combined score (e.g. security = 3×, code quality = 1×)
trust_tiers Path-based governance rules — which files need human sign-off vs. full AI autonomy
path_risk Mark paths as critical (always escalate) or safe (lower risk tier ceiling)
security_surface Glob patterns that flag files as security-critical, input-handling, data-access, config, or infra
file_roles Patterns to classify files as test, docs, generated, or build (affects triage)
privacy Compliance frameworks (GDPR, HIPAA, SOX, PCI-DSS) and data classification file path
severity_floor Per-risk-tier rules for suppressing low-value findings from the review output
validator Enable/disable the adversarial validator that challenges agent findings before display

Example

# review.yml — repo root
repo_risk_class: elevated

auto_approve:
  enabled: true
  blocked_target_branches:
    - main
    - release/*

weights:
  security_privacy: 4.0    # boost security weight
  test_quality: 3.0

trust_tiers:
  default_tier: spot_check
  rules:
    - tier: mandatory_human
      patterns: ["src/auth/**", "src/crypto/**"]
      reason: Security-critical code requires human review

path_risk:
  critical_paths:
    - pattern: "src/billing/**"
      reason: Payment processing — always escalate

security_surface:
  security_critical:
    - "**/auth/**"
    - "**/middleware/verify*"
  data_access:
    - "**/repositories/**"
    - "**/dal/**"
review.md Markdown

Free-form review guidelines written by your team. This is your voice — whatever matters to your project that the default agents wouldn't know.

The content is injected into every agent's system prompt under a Repository Review Guidelines section. Agents treat these as additional review criteria alongside their built-in expertise.

Example

# Review Guidelines

## Architecture
- All database access goes through the repository layer.
  Direct SQL in controllers is a hard no.
- New API endpoints must have an OpenAPI schema in /docs/api.

## Testing
- Service-layer changes need integration tests, not just unit tests.
- Mock the HTTP client, never the database.

## Security
- Never log PII fields (email, phone, SSN). Use the sanitizer.
- All new endpoints require auth middleware — no exceptions.

## Style
- Prefer explicit error returns over panic/throw in Go services.
- Keep functions under 50 lines. Extract helpers early.

Tip: Be specific and opinionated. Generic advice like "write clean code" doesn't help — the agents already optimise for that. Focus on rules unique to your project that an outsider wouldn't know.

How reviews work

When a PR is opened or updated, the pipeline runs these stages in order.

1

Discovery

Parses the diff and builds a rich context: languages, security surface, dependency graph, blast radius, change profile, hotspots. Loads review.yml and review.md from the repo.

2

Mechanical gates

Runs non-AI checks (linters, type checkers) to fail fast. If a gate fails the PR is hard-blocked immediately — no point running AI agents on code that won't compile.

3

Triage

Classifies the PR into a risk tier and selects which agents to run. Docs-only or generated-code PRs are marked trivial and skip AI entirely. Cross-stack changes, security surface touches, and new dependencies boost the tier upward.

4

AI agents (parallel)

Up to six specialist agents review the diff simultaneously, each through their own lens. Every agent returns a verdict (pass, warn, or flag-human) and a list of findings with severity and certainty levels.

Security & Privacy
Auth, injection, data leaks, compliance
Test Quality
Coverage, assertion quality, brittleness
Architecture & Intent
Design coherence, coupling, violations
Performance
Bottlenecks, resource waste, N+1s
Hotspot
Files with a history of bugs
Code Quality
Patterns, logging, maintainability
5

Scoring & decision

Each finding is scored by severity × certainty, then agent scores are combined using your configured weights. The decision engine cross-references the combined score, risk tier, repo risk class, and trust tier to choose the outcome.

6

Noise reduction

After scoring, findings go through three filters: cross-agent deduplication, severity floor (suppresses low-value findings by risk tier), and an adversarial validator that challenges each finding against the full diff. If this removes the evidence that triggered a reject, the decision is downgraded to human-review.

7

Post result

Posts a review comment with the decision, summary, and surviving findings. Sets the commit status, adds labels, and requests reviewers if human review is needed.

Decisions

Every review ends with one of four outcomes.

Auto-approve

Low risk, all agents pass. PR is approved automatically and the commit status is set to success. If the trust tier is spot-check, a human reviewer is also requested for an optional glance.

Human review

Score is borderline, agent flagged uncertainty, or the trust tier requires human sign-off. PR is not blocked but a reviewer is assigned. The most common escalation path.

Reject

At least one finding with high or critical severity, detected certainty, and a concrete actionable suggestion. PR receives a "changes requested" review. Merge is blocked until the issue is addressed.

Hard block

Mechanical checks failed or the combined score exceeded the hard-block threshold. Commit status is set to failure. Usually means the code doesn't pass basic gates.

Risk tiers

The triage stage assigns a risk tier based on what the PR changes. Higher tiers run more agents and need higher scores to auto-approve.

Tier Typical signals What runs
Trivial Docs-only, generated code, config-only No AI agents — auto-approved immediately
Low Small utility change, no risk signals Minimal agent set (typically 2–3 agents)
Medium Data layer, API boundary, or hotspot file touched Most agents run (typically 4–5)
High Security surface, new dependencies, architecture boundary crossed All 6 agents run

Amplifiers can boost the tier: cross-stack changes (Python + TypeScript), 3+ languages, SQL/Terraform/Docker files, elevated or critical repo risk class, and path-risk rules from review.yml.

Trust tiers

Separate from risk tiers. Trust tiers answer "who needs to approve this?" rather than "how risky is it?". Configured per-path in review.yml under trust_tiers.rules.

Tier Behaviour
ai_only Full AI autonomy. Auto-approve allowed, no human reviewer requested.
spot_check Auto-approve allowed, but a human reviewer is requested for an optional look. Default tier.
mandatory_human Human must approve. Overrides auto-approve regardless of score.
human_primary Specific reviewer group must approve (e.g. security team). Strictest tier.

Escalation: after agents run, trust can be bumped upward (never downward) if findings contain security keywords or high-severity issues. For example, a change to a utility file classified as ai_only might escalate to spot_check if an agent flags a potential auth issue.

Scoring

Each finding has a severity and a certainty. They combine into a numeric score that drives the decision.

Severity weights

Low1
Medium3
High6
Critical10

Certainty multipliers

Detected×1.0
Suspected×0.5
Uncertain×0.2

The combined score (0–10) is a weighted average of all agent scores. Default thresholds: auto-approve ≤ 4.0, human-review ≥ 4.0, hard-block ≥ 8.0. All configurable in review.yml.
