Concerns surfaced

The 401 retry branch is fighting your OAuth handler.

MEDIUM CWE-287 CWE-755 Infra

The Polly policy retries on HttpStatusCode.Unauthorized only when ReasonPhrase == "TenantThrottle". ReasonPhrase is optional in HTTP/2 and frequently stripped by load balancers — but the deeper issue is that GraphOAuthMessageHandler only attaches a token when Authorization is null. So a real 401 (expired token) loops three times with the same dead bearer and fails. Token refresh belongs in the handler, not in the retry policy.

Also affects: Infrastructure/Graph/GraphOAuthMessageHandler.cs — the handler caches but never invalidates on downstream 401.

Recommended fix
Fix sketch Drop the 401 branch from the Polly policy. In GraphOAuthMessageHandler.SendAsync, detect 401, invalidate the cached token, rebuild the auth header, and resend once. Keeps retry policy focused on transient HTTP errors and puts auth recovery where it actually has the cache to work with.

Tenant ID interpolated into a Graph URL without validation.

HIGH CWE-918 CWE-20 Services

GraphTenantClient.ListUsersAsync takes a tenant identifier from a request header and drops it straight into a URL template: $"https://graph.microsoft.com/v1.0/{tenantId}/users". The host is fixed, so this isn't classic SSRF, but a tenant value containing path segments (encoded or otherwise) can shift which Graph resource is hit. Defence-in-depth wants this guarded at our boundary, where we know what a tenant ID is supposed to look like.

The 429 retry ignores Microsoft's Retry-After hint.

LOW Infra

Standard exponential backoff (2^n seconds) is fine in isolation, but Graph returns Retry-After on 429s and ignoring it tends to make throttling worse — the same tenant ceiling is hit on every retry. A small WaitAndRetryAsync overload that reads the header and falls back to exponential when absent is the conventional fix.

Test coverage

Unit tests for the new Graph client and surrounding services. Two test files, ~600 LOC of happy-path coverage.

Tests 2 files · 797 LOC 1 concern
Concerns surfaced

OAuth handler has no tests for failure paths.

MEDIUM Tests

The new GraphOAuthMessageHandler has happy-path tests but no coverage for: (1) token acquisition failure, (2) cached-token expiry mid-request, (3) downstream 401 handling. Given the auth flow is the main risk surface flagged in the Microsoft Graph client capability above, missing failure-path tests means a regression here could ship undetected.

Also affects: Microsoft Graph client — the auth-retry concern is harder to verify a fix for without these tests.

Recommended fix
Fix sketch Add three test cases: (a) token provider throws → handler surfaces a typed exception; (b) cached token expired → handler refreshes before sending; (c) downstream returns 401 → handler invalidates cache and retries once.
