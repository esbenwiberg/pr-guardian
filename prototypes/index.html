PR Guardian · Design exploration

Alternative review modes

Two static prototypes exploring how to present a large PR's findings without the noise of the chapter view. Same example data (Microsoft Graph integration, 25 files, 23 findings) so the modes are directly comparable.

Prototype A

Decision Queue

One finding at a time, framed as a question with a single decision to make. Keyboard-driven (j/k nav, a/f/d/s act). 22 noise findings hidden, surfaced on demand. Empty queue = auto-approve.

 Prototype B

Narrative Briefing

Single scrollable page written as a briefing from the agent: what the PR is doing, three things worth your attention as prose, with diffs and fixes hidden behind inline expanders. 20 noise findings collapsed into an auditable footer.

 Prototype D

Capability Briefing (hybrid)

The PR breaks into capabilities (what the change does) rather than path-prefix chapters. Each capability collapses by default, expands to show concerns first, then files grouped by layer (Models / Services / Endpoints / Infra / Tests / Docs). Cross-capability concerns are pinned to their root cause with an "also affects" note. Clean capabilities show ✓ clean and stay collapsed.

 Prototype E

Capability Narrative (B + D mix)

D's capability scaffold with B's prose voice. Opens with a What / Why / How intro to set the story up, then each capability is a paragraph (not a structured concern card) with severity pills inline and diff/fix expanders attached to the relevant sentences. Layered file drill-in is collapsed at the bottom of each capability. Clean capabilities still get a one-paragraph summary so the reviewer knows the agent looked at them.
Scaffold decisions (locked): no pros section — clean capabilities just say ✓ clean; cross-capability concerns are pinned to root cause with an "also affects" mention; closed layer vocabulary (Models / Services / Endpoints / Validation / Infra / Tests / Config / Docs); fall back to mode B when surfaced findings ≤ 2 or capabilities = 1; triage (noise / fyi / decision) is a separate pipeline stage that all three modes consume.
Hosting: static files in /prototypes/. Open via file:// or any static server. A StaticFiles mount on the dev server is the next step once we pick a direction to take forward.
