What this PR is doing

This change adds a Microsoft Graph HTTP client to the platform — a typed IGraphService registered through DI, wrapped in a Polly retry pipeline, and authenticated by a new GraphOAuthMessageHandler that caches tokens. There is also new infrastructure plumbing (Infrastructure/Graph, Infrastructure/Services) and a pair of unit-test files covering the happy path. The shape is conventional and the registration site is idiomatic.

The risk surface is narrow but real. Two of the things that worry me sit inside the retry pipeline; the third is in the tenant-scoped Graph client. None of them are show-stoppers, but each is the kind of thing that's easier to fix now than after it ships.

What I'd flag for you

The 401 retry branch is fighting your OAuth handler.

MEDIUM CWE-287 CWE-755

The Polly policy retries on HttpStatusCode.Unauthorized only when ReasonPhrase == "TenantThrottle". That's a fragile match — ReasonPhrase is optional in HTTP/2 and frequently stripped by load balancers — but the deeper issue is that GraphOAuthMessageHandler only attaches a token when Authorization is null. So a real 401 (expired token) loops three times with the same dead bearer and fails. Token refresh belongs in the handler, not in the retry policy.

Recommended fix
Fix sketch Drop the 401 branch from the Polly policy. In GraphOAuthMessageHandler.SendAsync, detect 401, invalidate the cached token, rebuild the auth header, and resend once. That keeps the retry policy focused on transient HTTP errors and puts auth recovery where it actually has the cache to work with.

Tenant ID is interpolated into a Graph URL without validation.

HIGH CWE-918 CWE-20

GraphTenantClient.ListUsersAsync takes a tenant identifier from a request header and drops it straight into a URL template: $"https://graph.microsoft.com/v1.0/{tenantId}/users". The host is fixed, so this isn't classic SSRF, but a tenant value containing path segments (encoded or otherwise) can shift which Graph resource you're hitting. Microsoft's own validation isn't enough — defence in depth wants this guarded at our boundary, where we know what a tenant ID is supposed to look like (a GUID or a verified domain).

The 429 retry ignores Microsoft's Retry-After hint.

LOW

Standard exponential backoff (2^n seconds) is fine in isolation, but Graph returns Retry-After on 429s and ignoring it tends to make throttling worse, not better — you keep hitting the same tenant ceiling on every retry. A small WaitAndRetryAsync overload that reads the header and falls back to exponential when absent is the conventional fix here.
