Microsoft Graph integration with retry & OAuth handling

A briefing prepared by the review agent. The PR breaks down into four capabilities; two are clean, two carry concerns worth your attention.

esbenwiberg · 4 commits 37 files · +1,303 / −0 23 findings · 4 surfaced · 19 suppressed risk tier MEDIUM
What this PR does

It adds a first-class Microsoft Graph client to the platform — a typed IGraphService registered through DI, wrapped in a Polly retry pipeline, and authenticated by a token-caching GraphOAuthMessageHandler. Alongside that, it retires the legacy plugin-based Graph integration (six files deleted), updates the four endpoints that used it, and adds unit tests for the new client. A handful of unrelated agent-mode plumbing edits ride along.

Why it's here

The accompanying spec (specs/msgraph-di-consolidation.md) frames this as a consolidation: the legacy plugin pattern duplicated auth and retry logic across each Graph caller, so token expiry and rate limiting were handled inconsistently. The goal is to centralise both — one HTTP client, one retry policy, one auth handler — and treat Graph as an ordinary typed service. Most of the line count is mechanical: deletions in Legacy/Plugins/* and migrations in Legacy/Endpoints/*.

How it's structured

Standard layered shape. Infra registers the client and owns the cross-cutting handlers; Services hold the business-facing wrappers (MsGraphService, GraphTenantClient); Models hold the DTOs; Endpoints are updated to consume the new service. Tests cover the happy path. The risk surface is narrow but real, and concentrates inside the auth/retry interaction in the Infra layer — which is what most of this briefing is about.

Microsoft Graph client

3 concerns
11 files · 487 LOC · Models, Services, Infra

This is where the change earns its keep, and also where the risk concentrates. The shape is sound — a typed client behind DI, retry handled by Polly, auth handled by a dedicated DelegatingHandler — but two pieces of the implementation are pulling in opposite directions, and a third pulls against Microsoft's own throttling contract.

The first one is the auth/retry interaction. The Polly policy retries on HttpStatusCode.Unauthorized only when ReasonPhrase == "TenantThrottle" MEDIUM CWE-287 CWE-755. ReasonPhrase is optional in HTTP/2 and frequently stripped by load balancers — but the deeper issue is that GraphOAuthMessageHandler only attaches a token when Authorization is null. So a real 401 (expired token) loops three times with the same dead bearer and then fails. Token refresh belongs in the handler, not in the retry policy.

Recommended fix
Fix sketch Drop the 401 branch from the Polly policy. In GraphOAuthMessageHandler.SendAsync, detect 401, invalidate the cached token, rebuild the auth header, and resend once. Keeps the retry policy focused on transient HTTP errors and puts auth recovery where it actually has the cache to work with.

The second one sits one layer up. GraphTenantClient.ListUsersAsync takes a tenant identifier from a request header and drops it directly into a URL template HIGH CWE-918. The host is fixed, so this is not classic SSRF, but a tenant value containing path segments — encoded or otherwise — can shift which Graph resource is hit. Microsoft validates tenant IDs server-side, but our client should not rely on that for defence-in-depth; we know what a tenant ID is supposed to look like (a GUID or a verified domain), and we should enforce it at the boundary.

The last one is smaller and easy to forget. The 429 retry uses fixed exponential backoff and ignores Microsoft's Retry-After header LOW. That tends to make throttling worse, not better — the same tenant ceiling gets hit on every retry. A small WaitAndRetryAsync overload that reads the header and falls back to exponential when absent is the conventional fix here. Worth folding into the same change as the auth fix above.

Test coverage

1 concern
2 files · 797 LOC · Tests

Two test files, ~600 LOC of happy-path coverage for the new client. The coverage of the typed service methods is fine — call shape, deserialisation, basic error mapping all have cases. What is missing is the part that actually matters given the auth-retry concern in the Microsoft Graph client capability above: GraphOAuthMessageHandler has no tests for token-acquisition failure, cached-token expiry mid-request, or downstream 401 handling MEDIUM. If you ask for the auth fix above, ask for these tests as part of the same pass — they are the cheapest way to keep that fix from regressing.

Recommended fix
Fix sketch Add three test cases against GraphOAuthMessageHandler: (a) token provider throws → handler surfaces a typed exception; (b) cached token expired → handler refreshes before sending; (c) downstream returns 401 → handler invalidates cache and retries once.

My read

The shape is sound and the consolidation is the right call. Two of the four capabilities are clean and need nothing from you. The risk lives in the Microsoft Graph client — the auth-retry interaction is a real runtime bug under token expiry, and the tenant-ID interpolation is a small, easy hardening. Test coverage is coupled to the first fix: the missing failure-path tests are the cheapest way to keep the fix from regressing. I would ask for the auth change, the tenant-ID guard, and those three tests in one revision, and merge the Retry-After change as a follow-up if that pace is preferable.
