# ADR-010: Squashed migration baseline

## Status

Accepted - 2026-06-15.

## Context

The Alembic chain (`001`–`024`) had drifted from the SQLAlchemy models and could
not run end-to-end on an empty database:

- `003` re-added the three cost columns that `002` already creates
  (`DuplicateColumnError`).
- `014` created `scan_issues` with a foreign key to `scans`, but `scans` was not
  created until `019`.

Production never hit these because the schema source of truth is
`Base.metadata.create_all` — the app runs it on every startup (`init_db`), and
`docker-entrypoint.sh` bootstrapped legacy databases via `create_all` then
`alembic stamp 003`, so `001`–`003` never executed and `004+` ran against an
already-complete schema. The migrations were effectively vestigial and only the
"fresh empty database" path exercised the full chain — which was broken.

Fixing each ordering bug forward would mean editing several merged migrations
with no confidence the chain was finally consistent.

## Decision

Replace the `001`–`024` chain with a single baseline migration
(`001_baseline.py`) generated by `alembic revision --autogenerate` against the
models. The baseline's resulting schema was verified byte-for-byte (via
`pg_dump --schema-only`) to match `create_all`. Data seeds that had no runtime
equivalent are preserved in the baseline (the bootstrap admin from `008`); the
default-profile seed from `019` is not, because `storage.ensure_default_profile()`
creates it at runtime.

`docker-entrypoint.sh` re-stamps any existing database (detected by the presence
of the `reviews` table) to the baseline with `alembic stamp 001 --purge` — no DDL
runs against existing data — and runs the baseline from scratch only on a genuinely
empty database.

## Consequences

- `alembic upgrade head` works from an empty database again; fresh deploys and
  existing deployments converge on the same schema.
- The models remain the schema source of truth: the baseline equals `create_all`,
  and any divergence is a regression caught by re-running the schema diff.
- Future migrations are normal incremental revisions layered on `001`. The
  append-only/numbered convention resumes from `002`.
- Legacy artifacts that were never in the models (e.g. the `github_pats` table and
  its `uq_github_pats_single_default` index) are intentionally absent from the
  baseline. Existing databases retain them as harmless orphans; fresh databases
  never create them.
- Pre-baseline migration files are gone from the tree (still in git history), so
  one-time data-migration helpers they exported (the `github_pats` → connections
  backfill) and their tests are removed.
